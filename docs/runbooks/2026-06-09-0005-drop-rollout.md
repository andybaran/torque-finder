# Rollout runbook: deploy `feat/html-content-ingestion` + gated 0005 drop

Post-merge, manual procedure (spec §10, plan Task 13 Step 5 — adapted to the
mid-execution environment change). **Nothing here runs during plan execution.**
Each ⚠️ step requires an explicit user go-ahead.

## Current state (as of 2026-06-09)

- **Live DB:** Postgres in the Railway project **`manuals`** (to be renamed
  `parts-lookup`), public proxy `caboose.proxy.rlwy.net:31126`, 250 GB volume.
  The repo's local `.env` points at it. Alembic is at `0004_documents_chunks`
  (documents/chunks created and populated; legacy `pdfs`/`pages` intact).
- **Eval gate:** already green against this DB — **9 passed, 2 xfailed**
  (xfails tracked in issue #19).
- **OLD stack** (a Railway project on a *different* account): API
  `parts-api-production.up.railway.app` + Postgres
  `acela.proxy.rlwy.net:22358`, still at alembic `0003` with its own
  `pdfs`/`pages` intact. It keeps serving until cutover, then the user
  decommissions it. **Never run migrations against acela** — it is not part
  of this rollout and is retired as-is.
- The plan header's "production crash-loop window" does **not** apply to this
  rollout: the new service is born with an image that ships all migrations
  (0001–0005), and the old stack never sees 0004+.

## Procedure

### 1. Merge the PR

Squash-merge `feat/html-content-ingestion` into `main` (issue #2).

### 2. ⚠️ Deploy a NEW `parts-api` service in the `manuals` project

This is a **new service**, not a redeploy of the old one:

1. In the `manuals` Railway project, create a service `parts-api` from this
   repo's `main` (Dockerfile build — `railway.toml` already sets
   `builder = "DOCKERFILE"`).
2. Set variables in the Railway panel: `ANTHROPIC_API_KEY`, `VOYAGE_API_KEY`,
   `R2_ACCOUNT_ID` / `R2_ACCESS_KEY_ID` / `R2_SECRET_ACCESS_KEY` / `R2_BUCKET`,
   `OTEL_EXPORTER_OTLP_ENDPOINT` / `OTEL_EXPORTER_OTLP_HEADERS`, `SENTRY_DSN`,
   and `DATABASE_URL` using the project's **internal** Postgres host (not the
   `caboose` proxy).
3. On boot, the pinned `scripts/start.sh` runs
   `alembic upgrade 0004_documents_chunks` — a clean no-op (0004 is already
   applied) — and starts uvicorn. It deliberately does **not** reach 0005.

### 3. Smoke the new service (use its Railway-issued URL)

```bash
BASE=https://<new-parts-api-domain>
curl -s "$BASE/healthz"
curl -s "$BASE/docs" -o /dev/null -w '%{http_code}\n'   # expect 200
curl -s -X POST "$BASE/v1/query" \
  -H 'Content-Type: application/json' \
  -d '{"question": "What torque value applies to the component on page 28?"}' | python3 -m json.tool
```

Expect `"source_type": "pdf"`, `"torque"` containing `40 N-m`, a `source_url`
ending `#page=28`, and a non-null `screenshot_url`.

```bash
curl -s -X POST "$BASE/v1/query" \
  -H 'Content-Type: application/json' \
  -d '{"question": "In the SRAM Road AXS digital user manual, what torque do the crank arm bolts get?", "top_k": 5}' | python3 -m json.tool
```

Expect `"source_type": "html"`, a `https://docs.sram.com/...#...` `source_url`,
and `"screenshot_url": null`.

### 4. (Optional) re-run the eval gate

The gate is already green (9 passed, 2 xfailed → #19); re-run only for extra
confidence:

```bash
set -a && . ./.env && set +a
uv run --extra dev pytest tests/eval -m eval -q
```

Expected: **9 passed, 2 xfailed**.

### 5. ⚠️ Apply the 0005 drop (user-confirmed, against caboose only)

Only after the user confirms the smoke + eval are green. The `.env`
`DATABASE_URL` must point at the `manuals` Postgres
(`caboose.proxy.rlwy.net:31126`) — verify before running:

```bash
set -a && . ./.env && set +a
uv run alembic -c alembic.ini upgrade head
```

Expected output: `Running upgrade 0004_documents_chunks -> 0005_drop_pdfs_pages`.

A red eval instead leaves both schemas intact; if the drop was applied and
must be reversed, `downgrade()` rebuilds `pdfs`/`pages` from
`documents`/`chunks` (ids regenerated, content preserved).

### 6. Unpin `scripts/start.sh`

In a small follow-up PR/commit, restore the migration line to:

```sh
echo "[start] running database migrations (alembic upgrade head)..."
uv run --no-sync alembic -c alembic.ini upgrade head
```

(and remove the pin comment). Redeploy `parts-api`.

### 7. Re-smoke and cut over

Re-run step 3's curls after the unpinned deploy to confirm nothing depended on
the dropped tables. Then retire `parts-api-production.up.railway.app`: point
any consumers at the new service URL, and — when convenient — shrink/
decommission the old account's Railway project (API + acela Postgres). Do not
migrate or otherwise touch acela first; it is discarded wholesale.
